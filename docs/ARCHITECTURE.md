# 架构说明

## 设计目标

Daily Workbench 把“工作区”作为核心上下文。当前工作区身份、界面偏好、收件箱、任务、Markdown 笔记、今日日程、浏览器标签与收藏夹已经进入 SQLite，并由统一搜索与数据保护边界覆盖；终端会话在 Main 运行时按同一工作区边界隔离，但不伪装成可恢复的持久数据。业务模块不需要绕过进程隔离。

## 进程与信任边界

| 层             | 信任级别 | 职责                                        | 明确禁止                                   |
| -------------- | -------- | ------------------------------------------- | ------------------------------------------ |
| Electron Main  | 高       | 生命周期、窗口、PTY、浏览器会话、IPC 编排   | 将任意 Electron 对象直接交给页面           |
| Preload        | 高且受限 | 将窄、类型化的方法映射到 `window.workbench` | 暴露 `ipcRenderer` 或 Node.js 通用能力     |
| React Renderer | 可信 UI  | 布局、交互、状态展示、xterm 渲染            | 直接创建进程或读取本地文件                 |
| Remote Browser | 不可信   | 呈现用户访问的远程网页                      | preload、Node 集成、本地 IPC、默认设备权限 |

## 浏览器

右侧浏览器由 Main 管理多个 `WebContentsView`。React 只绘制标签栏、地址栏、收藏夹、下载托盘和占位区域，通过 `ResizeObserver` 取得内容区域的 DIP 坐标，再让 Main 更新活动视图边界。

- URL 只允许 `http:` 与 `https:`。
- 标签 URL、标题、活动标签和收藏夹按工作区持久化；每个工作区最多 12 个标签、500 个收藏夹。
- 只有活动标签会按需创建远程视图。切换工作区会销毁旧工作区全部远程视图及其监听器，随后从 SQLite 恢复新工作区的标签元数据，避免隐藏页面继续运行。
- 新窗口请求默认不创建额外 Electron 窗口；安全 URL 会在同一工作区的新标签中打开。
- 摄像头、麦克风、定位、通知、USB、蓝牙等权限默认拒绝。
- 所有远程标签使用独立的 `persist:workbench-browser` session；Cookie 和登录态因此在应用内共享，不承诺按工作区隔离。
- 面板关闭时隐藏视图。窗口关闭或应用退出时先通过窄 IPC 让可信 Renderer 完成未保存笔记确认；拒绝关闭不会触碰浏览器或数据库。确认响应发送前 Renderer 会同步冻结交互，响应失败则恢复；Main 收到确认后立即禁用并隐藏窗口，再停止接收新浏览器操作，等待进行中的切换与标签元数据写入，重新捕获最终 URL/标题并显式关闭 `webContents`。数据库只会在这些步骤完成后关闭，最终窗口使用已批准的强制销毁路径，不再触发第二次 `beforeunload`。
- 每次异步读取和 mutation 都绑定请求发起时的 `workspaceId`、标签 ID 与工作区 generation。切换工作区后的迟到结果会被拒绝，A→B→A 时 Renderer 仍以单调 revision 判定新旧快照。
- Main 在工作台、终端或远程页面取得键盘焦点时统一拦截非重复、非 IME 组合态的 `Ctrl/Cmd + N`，并向可信 Renderer 发出快速记录事件；远程内容仍得不到本地 IPC。
- 浏览器快捷键同时在可信 Renderer 和远程页面入口使用同一纯函数判定，拒绝 `keyup`、自动重复、IME 组合态和冲突修饰键。

## 下载

下载生命周期只存在于 Main，并只接受当前受管浏览器标签中由用户手势触发的 `DownloadItem`。

- 每个文件都使用系统保存对话框；Renderer 不能指定默认目录、保存路径或读取最终绝对路径。
- 同时最多保留 5 个活动下载；完成、失败或取消的记录按单调序号最多保留最新 100 条，裁剪会在发布快照前完成且只存在于本次运行。
- Renderer 只能按显式 `workspaceId` 和 opaque 下载 ID 请求暂停、恢复、取消、清理或定位文件；快照只暴露经过清理的文件名、来源主机、MIME、字节数和状态。
- 定位操作只允许已完成、仍存在的绝对普通文件，拒绝符号链接，并由 Main 调用系统文件管理器显示文件；不向远程内容或 Renderer 暴露通用路径打开能力。
- 非受管 `WebContents`、缺少用户手势、超出并发上限或字段校验失败的下载会在 `will-download` 阶段直接拒绝。
- 工作区归档提交成功后，Main 会取消该工作区仍活动的下载、移除其运行时记录并释放并发槽位；数据库归档失败时不提前清理。

## 终端

终端进程只存在于 Main，Renderer 只保留 xterm.js 的显示缓冲：

1. Main 提供固定、只读的 Shell Profile 能力快照。Windows 分别探测 PowerShell 7、Windows PowerShell、CMD 与默认 WSL；POSIX 探测系统默认 shell、Bash、Zsh 与 PowerShell 7。Renderer 不提交路径、参数、环境变量或 WSL 发行版名称。
2. WSL 探测只调用固定的 `wsl.exe --list --quiet`，带 3 秒超时、输出上限且不经过 shell。仅安装 `wsl.exe` 但没有发行版时 Profile 保持不可用。
3. 创建、激活、重启、输入、resize、清空和关闭都显式携带 `workspaceId` 与不可预测的会话 UUID。Main 同时核对当前工作区、不可变所有者和仍存活的会话身份。
4. 每个工作区最多 8 个会话，每个窗口最多 16 个。输出事件携带工作区、会话与单调序号，单次事件最多 64 KiB；Renderer 在 surface 尚未挂载时只保留有界的 256 KiB 尾部。
5. 每个会话使用独立 xterm surface 和 PTY。切换工作区会隐藏但保留后台会话；退出状态会保留标签供用户重启，显式关闭才移除标签。
6. 工作区归档成功提交后，Main 使目标会话身份失效并尽力终止全部 PTY；数据库归档失败时不提前清理。窗口关闭、Renderer 崩溃或应用退出也会幂等地清理全部会话，单个 `kill` 异常不会阻断其余资源。

终端状态、输出缓冲、CWD 和所选 Profile 只存在于本次运行，不进入 SQLite。启动 PTY 时保留平台所需系统环境变量，但不会把可执行文件路径或原始启动错误返回 Renderer。

## IPC 约定

共享协议放在 `src/shared`。命令使用 request/response，持续输出使用可取消订阅事件。所有 handler 都应同时验证：

- 调用者是否为可信主界面；
- 字符串长度、枚举值和数值范围；
- URL 协议、会话 ID 和边界坐标；
- 目标对象是否仍然存活。

工作区、收件箱、任务、笔记、日程、浏览器、搜索和终端请求必须显式带目标 `workspaceId`；搜索另接受固定的当前/全部活动工作区 scope。浏览器标签、收藏夹与下载操作还必须带对应 opaque ID，终端会话操作必须带对应 UUID。Main 不能在执行时重新读取“当前工作区”猜测归属。数据导入提交只接受短期 opaque ID 与预览摘要，不能携带路径。Main 主动触发快速记录、浏览器/终端状态更新、备份状态更新与地址栏聚焦只使用固定事件通道，不暴露通用事件订阅能力。

新增功能时，应先在共享层定义输入与输出，再实现 Main handler 和 Preload 映射，最后接入 Renderer。

## 数据库

SQLite 连接、迁移、Repository、搜索、备份策略和数据可移植性都只存在于 Main。应用从 `userData` 推导固定的数据、备份和导入隔离目录，Renderer 不能提供路径、SQL、迁移版本或备份原因。数据库管理只暴露类型化状态、受控备份/策略操作与逻辑数据包流程；工作区、收件箱、任务、笔记、日程与浏览器持久数据仍通过各自的窄业务接口访问。

数据库服务是应用级单例：首个窗口创建前打开；macOS 关闭单个窗口时保持连接，正常受控退出则先排空所有窗口的浏览器操作与元数据写入，再排空数据库操作队列并关闭。Windows 注销/关机和强制终止依靠 SQLite 事务与 WAL 做崩溃恢复，未完成的临时备份不会被列为可用备份。连接启用 foreign keys、WAL、busy timeout、defensive mode 与 `trusted_schema=OFF`，禁止加载扩展。

迁移通过构建期 raw import 嵌入 Main bundle，按连续版本在事务中执行，并把名称与 SHA-256 校验和写入历史表。已应用迁移发生漂移或数据库版本高于当前代码时拒绝启动。已有数据库升级前会先创建经过完整性验证的一致性备份；备份失败时不会继续迁移。

v7 追加外部内容 FTS5 索引和应用级备份策略/运行状态。逻辑导入不复用外部数据库表结构，而是在隔离区验证版本化数据包并构建当前 schema 的 staging 数据库；替换前必须生成并重新验证导入前备份，关闭连接后由 crash marker 驱动重启阶段的原子替换和失败回滚。成功安装会把旧数据库原子发布为 `pre-import-recovery-<id>.sqlite3`，作为不进入普通备份列表的最后恢复副本。备份列表仍不暴露任意路径恢复或删除。详细规则见[数据库与迁移](DATABASE.md)。

## 全局搜索

搜索 Service 与业务 Service 共享数据库单例和 FIFO 操作队列。v7 的外部内容 FTS5 索引覆盖收件箱正文、任务标题、笔记标题/正文、日程标题、浏览器标签与收藏夹标题/URL；结果始终回连原表检查活动工作区和归档状态。

- 查询在共享层做良构 Unicode、NFKC、长度与不可见字符校验，FTS 查询由 Main 构造为 phrase，Renderer 不能提交运算符。
- 两字符查询走参数化、转义后的 `LIKE`；三字符以上使用 trigram FTS，兼顾短中文查询与较长正文。
- scope 只允许当前工作区或全部活动工作区。排序先考虑精确/前缀/包含匹配和当前工作区，再使用类别、相关性、更新时间和稳定 ID。
- 每类及总结果都有固定上限，返回值显式标记截断类别；搜索结果只含导航所需的 opaque 身份、标题、受限摘要和工作区名称。

## 备份与数据可移植性

自动备份调度是应用级单例，不跟随工作区切换。策略默认关闭，只允许每日/每周、本地分钟、可选星期和 1–90 个保留数量；策略更新使用 revision。调度器为 single-flight，并持久记录最近尝试、成功 bucket、失败码和连续失败次数。创建与保留清理都完成后才记录当前 bucket 成功；任一步失败都会按 5 分钟起、最长 6 小时的指数退避重试完整备份轮次。保留数量在执行删除的同一个数据库队列操作中重新读取，不能用已被新设置取代的旧值删除文件。保留清理只作用于 `scheduled` 备份，手动、迁移前和导入前备份不会被自动清理。

逻辑数据包是版本化的规范 JSON 行流，不是 SQLite 文件，也不包含 FTS 索引或终端/浏览器会话运行时。Manifest 绑定源版本、记录计数和 SHA-256，解析器限制大小、记录数、字段集合和引用图。导出在文件对话框返回后立即冻结目标的“缺失或既有文件身份”，写入同目录唯一临时文件并 `fsync`；最终重命名前目标若被创建、替换或修改就拒绝覆盖，发布后再逐字回读并同步父目录。

导入预览在受控隔离目录中生成当前 schema staging 数据库，并以有时限的 opaque ID/摘要绑定确认；替换 marker 只记录受控文件名、摘要、阶段和导入前备份 ID，使启动恢复能够决定继续安装、提交或回滚。替换过程中每次文件发布和 marker 更新都同步文件与目录；缺失 marker 时只对“主数据库缺失且恰有一个受控 rollback”做自动恢复。

真正首装会在创建 `backups/` 和打开 SQLite 之前，先以文件与目录 `fsync` 发布固定的 `database-initializing-v1` 意图；只有这个有效意图可恢复首装崩溃留下的缺失或 0 字节主库。完整初始化后先耐久发布 `database-initialized-v1` 哨兵，再严格复核并删除意图、同步目录。哨兵存在、升级前版本留下的真实 `backups/` 目录存在、普通备份存在、retained recovery 存在或 rollback 状态不唯一时，主库缺失都不会被当成首装；意图不能绕过哨兵或已识别备份，预先存在但没有意图的 0 字节主库也直接拒绝。仅预建但完全空的应用数据目录仍可首次初始化。对既有数据库，Main 在 SQLite 打开前后复核同一文件身份、大小和修改时间，再允许运行 PRAGMA 或迁移。任何数据库/rollback 并存、多个孤儿、主数据库缺失但仅有 retained recovery、主库缺失而普通备份、旧版备份目录或哨兵仍存在，或主库路径被截断/替换的情况都会停止并保留现场，不创建一个空库伪装成首装。

数据目录在 POSIX 上限制为当前用户访问，启动时拒绝符号链接或错误文件类型；导入、导出和恢复也会在异步边界复核文件身份与内容摘要。这些机制覆盖崩溃、断电、意外同步和普通文件替换，但不把已能以同一账户持续改写应用私有目录的进程视为隔离对手：在最终系统调用之间主动交换整个目录 inode 需要平台专用的目录句柄/`openat` 方案，当前不作跨平台承诺。

## 工作区

工作区 Repository 和 Service 只使用数据库单例的同一连接，并复用关闭时等待的 FIFO 操作队列。Renderer 每次取得一个原子快照：活动工作区列表、当前工作区 ID 与当前偏好，不分别读取可能互相错位的状态。

- 新库在 Main 中生成一个默认工作区；UUID、时间戳和默认值不由 Renderer 提供。
- 创建会同时写入工作区、默认偏好和当前选择；任何一步失败都会整体回滚。
- 归档是软归档。不能归档最后一个活动工作区；归档当前项时会在同一事务中先切换到确定性的备用项。
- 当前页面、主题、侧栏、浏览器/终端面板开关和尺寸按工作区保存。
- Renderer 对偏好使用乐观显示、字段级 dirty 追踪和有序重试；工作区 mutation 返回的完整快照会按 workspace/revision 重基，不能覆盖其间已提交或待重试的偏好。
- 浏览器标签元数据、活动标签和收藏夹按工作区隔离；切换时销毁旧工作区远程视图，但共享的浏览器 session/Cookie/登录态仍为应用级上下文。下载记录只存在于 Main 运行时并按工作区过滤，归档工作区时会被清理。
- 终端进程、Shell、CWD 与缓冲区只存在于 Main 运行时并绑定创建时的工作区；切换会保留后台 PTY，归档成功会清理目标工作区，其他工作区不受影响。
- 已归档工作区的恢复和永久删除尚未暴露，数据仍会进入一致性备份。

## 快速记录与收件箱

收件箱与工作区共用同一个 SQLite 连接和 FIFO 操作队列。Renderer 取得包含 `workspaceId` 的完整活动条目快照，并按请求序号丢弃旧工作区或旧请求的迟到响应。

- 条目 ID、时间和撤销令牌只由 Main 生成；Renderer 只能提交目标工作区、正文和固定分类。
- 正文只去除首尾空白并验证良构 Unicode、控制字符和长度，不做 NFKC 改写，避免破坏 URL、代码和技术文本。
- 分类固定为未分类、任务线索、笔记和链接。“任务线索”不代表已经创建任务。
- 归档只写入 `archived_at`。成功提交后，Main 在内存中保存 15 秒有效的一次性 opaque 撤销令牌；后端有效期使用不受系统时钟回拨影响的单调时钟，令牌不能跨工作区、重复或过期使用，也不会在重启后恢复。
- 多条归档通知可以并存；切换工作区后旧响应不会写入新工作区界面。
- 工作区归档不会删除或改写其收件箱数据，但该工作区中的条目会变为只读，继续进入一致性备份。
- 转换为任务时，在单一事务中创建任务、写入唯一来源关系并归档收件箱条目；失败时整体回滚。已转换或已归档条目不能重复转换。

## 任务与 Today

任务与工作区、收件箱共用同一个 SQLite 连接和 FIFO 操作队列。Renderer 每次取得带 `workspaceId` 和 Main 计算的本地民用日期 `todayDate` 的完整任务快照；创建和转换只接受 `today` 或 `none` 计划意图，不允许页面伪造持久化日期、完成时间或来源关系。

- 任务 ID、时间戳、`plannedFor`、`completedAt` 和收件箱来源只由 Main 生成或推进。
- 状态固定为 `todo`、`in_progress`、`completed`；完成时间必须与状态一致。
- `plannedFor` 是本地民用 `YYYY-MM-DD` 或 `null`。Today 只展示当前工作区中计划到 `todayDate` 的任务，并与完整任务页共享同一真实快照。
- 标题可重命名，任务可加入或移出今日计划。所有 mutation 返回新的完整快照，Renderer 仍按工作区和请求序号处理迟到响应。
- `sourceInboxEntryId` 为可空且唯一的来源关系。显式转换会原子创建任务并归档来源收件箱条目；迁移到 v4 时不会把旧 `task` 分类自动转换为任务。
- 归档工作区保留任务和来源关系以进入一致性备份，但 Service 与数据库触发器都拒绝继续修改。
- Today 中的任务列表和进度与今日日程都使用真实 SQLite 数据。

## Markdown 笔记

笔记与既有业务模块共用数据库单例和 FIFO 操作队列。列表快照显式携带 `workspaceId`，所有创建、编辑、归档和收件箱转换都绑定请求发起时的目标工作区。

- 笔记 ID、时间戳、revision 和收件箱来源只由 Main 生成或推进。
- 标题最多 200 个 Unicode code point；Markdown 正文最多 100,000 个 code point。Main 会把 CRLF/CR 统一为 LF，保留 Markdown、代码和普通换行，同时拒绝 NUL、畸形 Unicode 和不支持的控制字符。
- 编辑与归档必须提交 `expectedRevision`。数据库中的 revision 已变化时拒绝旧写入，Renderer 需要基于最新快照重新处理，不能让迟到自动保存覆盖新内容。
- 归档是软归档。归档工作区中的笔记继续进入备份，但 Service 与 Trigger 都拒绝修改或删除。
- “笔记”分类仍只是收件箱处理意图。显式转换会在一个事务中归档来源、创建真实笔记并建立唯一来源关系；同一来源不能同时转换为任务和笔记。
- Markdown 内容始终按不可信文本处理。预览只通过受控 React 节点生成，不执行原始 HTML，也不把 `javascript:`、`data:` 或本地文件链接带入可信 Renderer。

## 今日日程

日程快照包含目标工作区、Main 计算的本地 `todayDate` 和当天活动项。当前阶段只处理“今天”，不包含重复规则、提醒、系统通知或外部日历同步。

- 日程 ID、持久化日期、时间戳和 revision 由 Main 控制；Renderer 只能提交 `expectedDate`、标题、固定类型以及分钟范围。
- 类型固定为 `focus`、`meeting`、`review`、`personal`。时间使用本地民用日期和 `0..1440` 分钟边界，结束分钟必须晚于开始分钟，不依赖 locale 字符串或 UTC 日期切片。
- 创建、编辑和归档都会验证 `expectedDate` 与 Main 当前日期一致；跨午夜后旧页面不能把昨天的请求写入今天。
- 编辑与归档使用 `expectedRevision` 防止迟到响应覆盖新状态。持久化日期不可修改，跨日移动留给未来独立功能。
- Renderer 在本地午夜、窗口重新聚焦和页面重新可见时丢弃旧日快照并重新读取，覆盖系统睡眠和时区调整后的日期变化。
- 归档工作区保留日程历史以进入一致性备份，但不再允许修改或删除。

## 浏览器标签与收藏夹

浏览器持久数据与既有业务模块共用数据库单例和 FIFO 操作队列。首次读取活动工作区时，Main 会在一个事务内初始化默认标签和活动状态；Renderer 不提供 ID、时间戳、revision 或页面标题。

- 标签、活动标签状态和收藏夹严格绑定 `workspaceId`。所有 mutation 返回完整持久快照，Main Controller 再合并当前远程页面的前进/后退、加载状态和仅运行时下载。
- 浏览器状态使用精确 `revision + 1` 触发器。关闭最后一个标签会原子重置为默认页，保证活动工作区始终至少有一个标签和有效活动引用。
- 标签导航完成后的 URL 与标题按每标签合并到单一有界写入 worker，连续事件只持久化最新期望值；显式导航从 Main 接受后会保留 pending URL，直到真实导航成功或失败，切换/退出期间不会用仍未提交的旧 `getURL()` 反写。关闭、收藏、工作区切换或受控退出前会等待同一标签的元数据落盘，避免写放大、收藏旧 URL、删除后迟到写入或关窗丢失最终状态。
- 收藏夹只接受当前受管标签的安全 HTTP(S) URL；同一工作区 URL 唯一。归档工作区中的标签、状态和收藏夹保留进备份，但 Service 与数据库触发器拒绝继续修改或删除。
- SQLite 不保存网页会话历史、Cookie、登录态或下载记录；这些运行时信息不会伪装成工作区持久数据。

## 打包

Electron Forge 负责启动、原生模块 rebuild 与平台打包；Vite 分别构建 Main、Preload 和 Renderer。通过项目 npm scripts 启动或打包前，会清理明确的 `.vite` 与 `out` 目录，防止旧入口或旧安装包混入产物。`node-pty` 保持外部依赖，并由 auto-unpack-natives 从 ASAR 解包。

应用启用 Electron fuses，关闭 Node CLI 参数、`NODE_OPTIONS` 和浏览器专用 V8 snapshot 等非必要入口。Windows ConPTY 的关闭流程依赖 `child_process.fork`，因此必须保留 RunAsNode。可信 Renderer 当前通过 `file://` 加载，所以暂时显式保留该协议的额外权限；迁移到自定义 `app://` 协议后再关闭。

Electron 43 有 9 项 V1 fuse，而 Forge 7 当前兼容的 `@electron/fuses` 1.x 只能命名前 8 项。打包验证会直接断言 wire 长度和全部 9 项状态（包括 `WasmTrapHandlers`）；Electron 新增 fuse 或任一状态漂移都会使 CI 失败。Windows CI 还会对 Squirrel 构建同时产生的未打包应用负载执行原生文件检查、多终端生产管理器，以及工作区、收件箱、任务、笔记、日程、浏览器 v7 持久数据、v0–v6 迁移、FTS 回填和备份冒烟；ASAR 门禁同时检查浏览器、下载与终端 IPC 安全入口和已移除旧协议，成功后才生成校验和并上传安装制品。最终 NUPKG 负载复验仍由独立的 Issue #9 跟踪。

下载运行时另有独立 Vite 入口，只导入生产 `DownloadManager`，不会给产品 Main 增加测试开关。Linux 在 Xvfb、Windows 在固定的 Electron 43.2.0 可执行文件中，用本地 HTTP Range 服务和真实 `WebContentsView` 用户手势验证 `will-download`、SHA-256、暂停/恢复、取消、非受管/无手势拒绝及定位回调；ASAR token 门禁继续负责证明相同安全入口已进入正式包。

终端运行时也有独立 Vite 入口，只导入生产 `TerminalManager` 与 Profile resolver。Linux 与 Windows 都在正式打包负载中验证固定 Profile 发现、两个并行会话、独立输出、resize、clear、关闭一个后另一个继续工作，以及 A→B→A 工作区隔离；只有机器真实存在默认 WSL 发行版时才执行 WSL 会话断言，否则必须报告该 Profile 不可用。

原生模块不能跨系统复用。Windows 安装包必须在 Windows x64 环境执行 `npm ci` 与 `npm run make`，并在升级 Electron 或 `node-pty` 后验证打包产物中的真实终端。

完整依赖树与生产依赖采用两层门禁：生产依赖不允许任何等级的已知漏洞；Forge 尚未修复的开发期链路按 GHSA 设置有期限例外，任何新漏洞、例外过期或风险进入生产树都会阻断。详见[依赖风险说明](DEPENDENCY_RISKS.md)。

## 业务数据边界

业务 Repository 会继续放在 Main，Renderer 仍只能经 preload 使用业务 API：

```text
Workspace
├─ Inbox ✓
├─ Tasks ✓
├─ Notes ✓
├─ Schedule ✓
├─ Focus sessions
├─ Browser tabs/bookmarks ✓
├─ Terminal sessions/profiles ✓ (runtime)
├─ Global search/quick actions ✓
├─ Backup policy/data portability ✓
└─ Layout state ✓
```

后续迁移和数据管理扩展仍需沿用同一边界，避免页面直接依赖表结构或文件路径。
