# 数据库与迁移

Daily Workbench 使用 Electron 自带的 `node:sqlite`。数据库只在 Main 进程中打开；Renderer 不能接触 SQL、数据库文件路径或任意备份目标。

## 目录与生命周期

Main 从 `app.getPath('userData')` 推导受控的数据目录：

```text
userData/
└─ data/
   ├─ daily-workbench.sqlite3
   └─ backups/
      └─ daily-workbench-*.sqlite3
```

在 POSIX 系统上，应用会把既有或新建的 `data/` 与 `backups/` 目录收紧为仅当前用户可访问的 `0700`，数据库与已完成备份使用 `0600`。Windows 依赖用户配置目录的系统 ACL；当前阶段不自行重写 ACL。

数据库服务是应用级单例，在第一个窗口创建前打开。macOS 上关闭并重新打开窗口不会重复创建连接；正常受控退出先等待可信 Renderer 对未保存内容给出明确许可，拒绝时浏览器、终端与数据库保持可用。确认传输期间 Renderer 会冻结交互且在传输失败时恢复；Main 收到确认后立即禁用并隐藏窗口，再阻止新浏览器/终端操作，等待浏览器元数据排空并幂等终止窗口中的全部 PTY，然后停止数据库新操作、等待正在进行的备份、checkpoint WAL 并关闭连接，最后强制销毁已批准的隐藏窗口。Windows 注销、关机或进程被强制终止时不能依赖 Electron 的退出事件，此时由 SQLite 事务/WAL 保证崩溃恢复，未完成的 `.partial` 备份不会进入可用备份列表。

本阶段的终端会话、Shell Profile、CWD 与输出缓冲均为 Main 运行时状态，不进入 SQLite，因此数据库迁移仍停留在 v6；应用重启不会恢复终端，也没有默认 Profile 或自定义命令的持久字段。

所有页面只能调用以下窄接口：

- 获取数据库健康状态；
- 在应用控制的目录中创建手动备份；
- 列出由应用生成的最近备份。
- 获取工作区原子快照；
- 创建、重命名、切换或归档受控工作区；
- 以严格字段 patch 更新指定活动工作区的界面偏好。
- 获取指定活动工作区的收件箱快照；
- 创建、分类和软归档收件箱条目；
- 使用 Main 签发、由单调时钟判定有效期的一次性短期令牌撤销刚完成的归档。
- 获取指定活动工作区的完整任务快照及 Main 计算的本地 `todayDate`；
- 创建或重命名任务，更新固定状态，并通过 `today`/`none` 意图加入或移出今日计划；
- 在单一事务中把收件箱条目转换为带唯一来源关系的任务。
- 获取指定活动工作区的 Markdown 笔记快照；
- 创建、按 revision 更新或软归档笔记，并在单一事务中把收件箱条目转换为真实笔记；
- 获取指定活动工作区由 Main 计算日期的今日日程；
- 创建、按 revision 更新或软归档当天的固定类型时间段。
- 获取指定活动工作区的浏览器标签、活动标签状态和收藏夹；
- 新建、激活或关闭标签，持久化受控 URL/标题，并切换或移除收藏夹。

接口不接受路径、SQL、迁移版本或备份原因。恢复、导入、删除备份不属于当前阶段。

## 连接策略

数据库适配层统一启用以下约束：

- foreign keys；
- WAL journal；
- `synchronous=NORMAL`；
- 5000ms busy timeout；
- `trusted_schema=OFF`；
- defensive mode；
- 禁止扩展加载与双引号字符串兼容行为。

Repository 负责 SQL 与严格行映射，Service 负责生命周期、串行化、事务、不变量和对外返回值。业务页面只能通过 `Renderer → Preload → 可信 IPC → Service → Repository` 接入。数据库元数据与工作区、收件箱、任务、笔记、日程、浏览器标签和收藏夹的启动校验在同一事务提交，校验失败不会留下已更新的 `last_opened_at` 半状态。

工作区、收件箱、任务、笔记、日程或浏览器事务如果无法可靠回滚，数据库服务会立即进入 fatal 状态：当前错误向上返回，已排队及后续读取、写入和备份全部拒绝，直到连接关闭并由新实例重新打开。应用不会在事务结果未知的连接上继续服务。

## 迁移规则

迁移源文件放在 `migrations/`，通过 `?raw` 在构建时嵌入 Main bundle，避免安装后依赖源码目录。

1. 版本必须从 1 开始连续递增。
2. 已合并或发布的迁移禁止修改、重排或删除，只能追加新文件。
3. 每条已应用迁移记录版本、名称、SHA-256 校验和与应用时间。
4. 启动时会比较代码中的迁移与数据库历史；缺号、未知版本、名称或校验和漂移都会阻止打开。
5. 待执行迁移在 `BEGIN IMMEDIATE` 事务内应用；任何错误都会回滚本次启动中的全部待执行迁移。
6. 迁移 SQL 禁止自行提交、开启保存点、执行 `PRAGMA` 或 `ATTACH/DETACH` 外部数据库；事务、版本、连接安全设置和数据库路径只由迁移框架控制。
7. 迁移成功后必须通过 quick check、foreign-key check，并重新核对 WAL、foreign keys、`synchronous=NORMAL`、5000ms busy timeout 与 `trusted_schema=OFF`。

已有数据库需要升级时，应用会先生成 `pre-migration` 一致性备份；备份创建或校验失败时不会执行迁移。新建空数据库不会生成无意义的迁移前备份。

当前迁移版本：

- `0001_database_foundation.sql`：应用元数据与迁移底座；
- `0002_workspaces.sql`：工作区、工作区偏好、单例当前选择、活动名称约束和归档保护触发器。
- `0003_inbox.sql`：按工作区隔离的收件箱条目、固定分类、软归档索引和归档工作区写保护。
- `0004_tasks.sql`：按工作区隔离的任务、固定状态、今日计划、唯一收件箱来源和归档工作区写保护。
- `0005_notes_schedule.sql`：按工作区隔离的 Markdown 笔记与今日日程、revision 并发保护、收件箱唯一来源和软归档保护。
- `0006_browser_tabs_bookmarks.sql`：按工作区隔离的浏览器标签、活动标签状态与收藏夹、数量上限、revision 和归档工作区写保护。

工作区 v2 始终保证至少一个活动项、一个有效当前选择以及每个工作区一条偏好。归档保留工作区和偏好行；当前阶段不提供恢复、永久删除或 Renderer 自定义归档时间。名称在 Main 中执行 Unicode NFKC 规范化并生成唯一键，颜色只接受内置调色板。

从原型版本首次进入 v2 时，Renderer 会严格读取旧 `localStorage` 中合法的页面、主题和布局值，并只在数据库仍为单一默认工作区时导入；只有 SQLite 写入成功后才清理旧键。旧版硬编码工作区选择不会被当作真实身份导入，写入失败时会保留可重试的 dirty patch 和旧键。

收件箱 v3 不导入界面中的演示条目，也不会把原型 `localStorage` 任务当作真实收件箱数据。条目允许重复正文，分类只接受 `uncategorized`、`task`、`note`、`link`；其中 `task` 只是处理意图。归档工作区中的条目原样保留，但触发器和 Service 都拒绝继续修改。

任务 v4 也不会导入 `daily.today.tasks` 中的演示任务，或自动转换既有 `task` 分类收件箱条目。任务状态固定为 `todo`、`in_progress`、`completed`；`planned_for` 只保存 Main 计算的本地民用 `YYYY-MM-DD` 或 `NULL`，完成状态与 `completed_at` 必须一致。Renderer 只能提交 `today`/`none` 计划意图，不能伪造日期、ID、时间戳或来源关系。

显式收件箱转换在一个事务内创建任务、写入唯一 `source_inbox_entry_id` 并软归档来源条目。任一步失败都会整体回滚；同一来源不能转换两次。归档工作区中的任务和来源关系继续进入备份，但不能再修改。

笔记 v5 不导入 Renderer 中的演示卡片，也不会因为收件箱分类为 `note` 就自动创建笔记。标题最多 200 个 Unicode code point；正文最多 100,000 个 code point，并把 CRLF/CR 统一为 LF。正文允许 Markdown、普通换行和 Tab，但拒绝 NUL、畸形 Unicode 与不支持的控制字符。创建时 revision 为 1；更新和软归档必须携带当前 `expectedRevision`，并在数据库内只递增一次。已归档笔记和归档工作区笔记不可修改，笔记不提供永久删除。

显式“收件箱转笔记”在一个事务内归档来源、创建笔记并建立唯一来源关系。任务与笔记共享同一个来源排他边界：同一收件箱条目不能同时转换为任务和笔记，已关联来源不能恢复。转换任一步失败都会整体回滚。

日程 v5 只保存当天的真实时间段，不包含重复、提醒、通知或外部日历同步。类型固定为 `focus`、`meeting`、`review`、`personal`；`scheduled_for` 是 Main 计算的本地民用 `YYYY-MM-DD`，时间范围使用 `start_minute`/`end_minute`，满足 `0 <= start < end <= 1440`。Renderer 提交的 `expectedDate` 必须与 Main 当前日期一致，不能伪造持久化日期；更新和软归档也必须携带当前 revision。日程日期不可原地修改，归档项和归档工作区日程不可再修改，日程不提供永久删除。

Today 的任务、进度和日程都使用这些真实行。Renderer 会在本地跨午夜、窗口重新聚焦或页面重新可见时丢弃旧日快照并重新读取，避免系统睡眠或时区变化后继续显示或写入昨天的数据。

浏览器 v6 不导入旧版仅运行时的 URL、历史或页面会话。首次访问活动工作区时，Service 会在单一事务中创建一个默认标签和对应的 `browser_workspace_state`；活动工作区始终至少保留一个标签，关闭最后一个标签会原子重置该行。每个工作区最多 12 个标签和 500 个收藏夹，收藏 URL 在工作区内唯一，只允许受控 HTTP(S) 地址。

浏览器标签、状态和收藏夹的 ID、时间戳与 revision 都由 Main 生成或推进。状态更新必须严格执行 `revision + 1`，标签与状态时间不能回拨，启动和备份还会验证状态时间不早于同工作区的标签或收藏夹；标签工作区不可改写，活动标签必须引用同工作区现存标签。远程页面标题会移除控制字符和双向文字覆盖/隔离符后再持久化，防止标签视觉欺骗。高频 URL/标题事件在 Main 中按标签合并为有界写入，状态切换与受控退出会排空最终期望值。归档工作区的全部浏览器持久数据继续进入备份，但 Trigger 与 Service 都拒绝继续写入或删除。Cookie、登录态、网页前进/后退历史和下载记录不进入 SQLite。

## 备份与未来恢复

备份使用 SQLite online backup API 写入同目录的唯一临时文件，验证快照可只读打开且通过迁移、工作区以及对应 schema 版本的全部收件箱、任务、笔记、日程、浏览器标签和收藏夹行映射后，再原子重命名为最终文件。列表会忽略临时文件、旁路文件和符号链接；Renderer 得到的是序列化元数据，不是绝对路径。

当前版本不会自动删除备份，也不会提供恢复 IPC。未来恢复功能必须在独立变更中完成以下保护：

1. 只接受列表中的 opaque ID，不接受任意路径；
2. 校验应用标识、迁移历史、schema 版本、quick check 与 foreign-key check；
3. 替换前自动创建恢复前快照；
4. 关闭连接并在重启阶段原子替换数据库；
5. 使用 crash marker，在重新打开或迁移失败时自动回滚。

## 验证矩阵

- 普通 Node 测试：新建、v1/v2/v3/v4/v5 升级、迁移回滚、工作区/收件箱/任务/笔记/日程/浏览器事务、撤销令牌、任务与笔记转换、revision 冲突、工作区隔离、数量上限、损坏拒绝、备份一致性和关闭队列。
- Linux 打包测试：真实 Electron 可执行文件中的 `node:sqlite`、WAL、新库与 v1/v2/v3/v4/v5→v6、工作区、收件箱、任务、笔记、日程、浏览器标签/收藏夹、两类转换、备份和重开；同一正式负载另运行生产多终端管理器冒烟。
- Windows 打包测试：同一数据库、工作区、收件箱、任务、笔记、日程与浏览器 v6 冒烟链路，以及生产多终端管理器，在 Squirrel 构建产生的未打包应用负载所对应的 Electron/SQLite/ConPTY 运行时中通过。最终 NUPKG 负载复验由独立的 Issue #9 跟踪。

开发机 Node 与 Electron 内置 Node/SQLite 的小版本并不完全相同，因此普通单元测试不能替代打包后冒烟测试。
